SESSION 009 — F196 (perception-daemon)  ·  2026-07-17  ·  model: claude-opus-4-8
-------------------------------------------------------------------------------
FEATURE: F196 — the perception pipeline runs as DECOUPLED MULTI-RATE LOOPS
(capture → inference → smoothing/metrics → render), each at its own independent
rate, so slow inference never blocks rendering. Second of the F195-F203 daemon
loops, built directly on F195's Web-Worker inference path.

WORK:
- frontend/src/app/pose/rate-meter.ts (NEW): a ~30-line sliding-window RateMeter.
  Counts tick()s inside a trailing 1s window and reports Hz; old ticks age out so
  a stalled loop decays toward 0 (a slow loop genuinely reads lower than a fast
  one — no stale peak). Dependency-free (Karpathy ethos).
- frontend/src/app/pose/pose-engine.ts: the engine now ticks an inferenceMeter on
  every worker 'result' message and exposes inferenceRate(now) — the true (lower)
  inference throughput, sampled by the metrics loop.
- frontend/src/app/capture/capture.ts: split the single rAF loop into THREE
  decoupled loops running at their own rates:
    • L1 Capture — driven by requestVideoFrameCallback (per decoded camera frame,
      ~15-30Hz; ~30Hz setTimeout fallback). Hands frames to the worker; the engine
      drops frames while one is in flight so the worker's lower rate never stalls
      capture.
    • L5 Render — a requestAnimationFrame loop (~display rate) that ONLY draws the
      latest worker skeleton. Never awaits inference → stays high (~120Hz headless).
    • L4 Metrics — a ~10Hz self-timed setTimeout loop, decoupled from render, that
      samples every loop's RateMeter and publishes captureHz/inferenceHz/metricsHz/
      renderHz signals (future F199/F200 findings hang off this same loop).
  ngOnDestroy tears down rAF + both timers.
- capture.html: a live perf panel (data-testid perf-panel) with per-loop readouts
